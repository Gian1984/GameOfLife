4IPO3 : Devoir des Rois Mages (Q54)

Enoncé du groupe POOO

But

Convertir une application procédurale en application orientée objet.

Thème : le jeu de la vie

Points pédagogiques clés :
  • Attributs et méthodes privés, avec accesseurs éventuels.
  • Aggrégation : Le jeu est un ensemble d'objets.
  • Composition : La grille est un ensemble de cellules.
  • Héritage et Polymorphisme : Il peut y avoir différents types de cellule à implanter, enfants d'une classe abstraite.
  • Design Pattern : Facture, Iterator, Singleton, Observer sont implantables.
  • Graphical User Interface & programmation événementielle

Consignes
  1. Fichiers à rendre

    • liveview.py : avec toutes les classes 'view'
    • livemodel.py : avec toutes les classes 'model'
    • livecontroller.py : avec toutes les classes 'controller'
    • livecounter.py : avec le compteur éventuel
    • main.py : le code nécessaire uniquement pour lancer l'application (le controlleur)

    Remarque : Il peut y avoir plusieurs classes par fichier py.

  2. Mvc

    • Développer ce jeu de la vie en POO sous architecture MVC, telle que vue au cours.
    • Implanter dans LiveModel une matrice de cellules avec les indices standards (0, 1, 2, 3 ...). De son côté, LiveView peut conserver la matrice pixelisée du code de départ (indices 0, 10, 20, 30, ...).

  3. Interface utilisateur

    • L'utilisateur peut modifier la couleur des cellules (vivante/morte) ou de la grille.

  4. Canvas

    • Implanter l'itération de sortie (affichage de la matrice) avec le design pattern Iterator appliqué à l'objet nécessaire (par ex. LiveCanvas).

  5. Stratégie

    • Proposer au départ une 'configuration aléatoire' : 25% des cellules vivantes au départ, dispersées aléatoirement.
      Indice : s'inspirer de canon, càd introduire à côté du bouton 'Canon' un autre bouton 'Aléa'.
      Indice : classe LiveModel
      Bonus : appliquer design pattern strategy

  6. Fonctionnalités supplémentaires

    • Afficher les cellules avec des couleurs différentes suivant le nombre de voisins.

Evaluation

L'évaluation de votre travail comprend d'une part la validation du bon fonctionnement de votre application (suivant les consignes), et d'autre part, votre manière de coder la POO. Bien coder signifie :

  • Points d'attention :
    • attributs et méthodes privés dans la mesure du possible
    • utilisation de composition, de dictionnaire ou de liste d'objets
    • héritage
    • couplage minimal entre objets, modularité
    • design pattern
  • L'ordonnancement des attributs et méthodes dans une classe est la suivante :
    1. attributs de classe
    2. méthodes de classe (@classmethod)
    3. méthodes de création d'objet (singleton, factory, __new__)
    4. constructeur (__init__ avec définition de tous les attributs d'instance)
    5. destructeur éventuel (__del__)
    6. autres méthodes magiques (__str__, __next__, ...)
    7. accesseurs
    8. méthodes publiques
    9. méthodes protégées
    10. méthodes privées
  • Au sein des classes, ne jamais faire appel à une variable globale, notamment celles définies dans le main. Toujours passer les variables nécessaires par argument de méthodes.
  • Tous les attributs sont privés ou protégés. Certains peuvent avoir des accesseurs (get et/ou set). Dans ce cas, utilisez les décorateurs appropriés.
  • Toutes les méthodes sont privées ou protégées, sauf celles qui ont besoin d'être publiques.
  • Utiliser les méthodes magiques de Python (__str__, etc.) chaque fois que c'est justifié.
  • Utiliser les design pattern chaque fois que c'est justifié.
  • Prévoir éventuellement un petit test unitaire pour les classes développées.
